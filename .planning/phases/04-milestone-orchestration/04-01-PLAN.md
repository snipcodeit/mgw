Create the milestone.md slash command file following the exact MGW command pattern (see run.md and project.md for structure reference). The command must:

**Frontmatter:**
```yaml
---
name: mgw:milestone
description: Execute a milestone's issues in dependency order — auto-sync, rate-limit guard, per-issue checkpoint
argument-hint: "[milestone-number] [--interactive] [--dry-run]"
allowed-tools:
  - Bash
  - Read
  - Write
  - Edit
  - Glob
  - Grep
  - Task
  - AskUserQuestion
---
```

**Objective section:** Orchestrate execution of all issues in a milestone by delegating each to `/mgw:run` in dependency order. Sequential execution, autonomous by default. Handles resume, failure cascading, rate limits, and milestone completion.

**Execution context:** Reference all 4 workflow files (state.md, github.md, gsd.md, validation.md).

**Process steps:**

`<step name="parse_arguments">`: Parse $ARGUMENTS for optional milestone number and flags (--interactive, --dry-run). If no milestone number, read `current_milestone` from project.json.

`<step name="validate_and_sync">` (MLST-03): Run validate_and_load from state.md. Then run batch staleness check (from state.md) to sync all active issues. Non-blocking — if sync fails, log warning and continue.

`<step name="load_milestone">`: Load project.json, extract the target milestone's issues array. Validate milestone exists and has issues. Display milestone name and issue count.

`<step name="resolve_execution_order">`: Implement topological sort (Kahn's algorithm) inline with python3. Use `depends_on_slugs` from project.json issues to build the dependency graph. Resolve slug-to-github_number mapping. Detect cycles — if found, error with cycle details and refuse to proceed. Break ties by `phase_number` (lower first). Filter out issues with `pipeline_stage == 'done'`. Mark issues blocked by failed issues as blocked.

`<step name="rate_limit_guard">` (MLST-04): Check `gh api rate_limit` remaining calls. Estimate 25 calls per issue. If remaining < estimated need: calculate how many issues can safely run, display warning with count and reset time. Cap the loop at safe count. If check fails (network), log warning and proceed.

`<step name="dry_run">`: If --dry-run flag: display execution plan table (issue order, dependency graph, status, estimated calls) and exit without executing. Table format:
```
Execution Plan: Milestone {N} — {name}
| Order | Issue | Title | Status | Depends On | Blocks |
|-------|-------|-------|--------|------------|--------|
```

`<step name="resume_detection">`: Check for issues with `pipeline_stage` not in ('new', 'done', 'failed'). For each in-progress issue: check if worktree exists (`git worktree list | grep issue/${NUMBER}`). If worktree exists, clean it up (`git worktree remove`). Reset pipeline_stage to 'new' for that issue (restart from scratch per CONTEXT.md decision).

`<step name="execute_loop">` (MLST-01, MLST-05): Sequential loop over sorted issues. For each issue:
1. Check if issue is blocked by any failed issue — if so, skip, mark dependents blocked too
2. Display: `Running issue #${N}...`
3. Quick check: `gh issue view ${N} --json state -q .state` — if not 'OPEN', skip
4. Check rate limit remaining — if < 25, stop loop with message
5. Spawn `/mgw:run` via Task():
```
Task(
  prompt="
    <files_to_read>
    - ./CLAUDE.md (Project instructions — if exists, follow all guidelines)
    - .agents/skills/ (Project skills — if dir exists, list skills, read SKILL.md for each, follow relevant rules)
    </files_to_read>

    Run the full MGW pipeline for issue #${ISSUE_NUMBER}.
    Read /home/hat/.claude/commands/mgw/run.md for the complete workflow.
    Follow ALL steps: validate_and_load, create_worktree, post_start_update,
    execute_gsd_quick or execute_gsd_milestone, create_pr, cleanup_and_complete.
  ",
  subagent_type="general-purpose",
  description="Run pipeline for #${ISSUE_NUMBER}"
)
```
6. On success: update project.json `pipeline_stage = "done"` (state.md pattern), display "Done."
7. On failure: update project.json `pipeline_stage = "failed"`, add `pipeline-failed` label (`gh issue edit ${N} --add-label pipeline-failed`), post failure comment with collapsed milestone progress table, display "Failed.", mark all dependent issues as blocked
8. If --interactive flag: pause with AskUserQuestion after each issue: "Continue to next issue?" with options Continue/Skip next/Abort
9. Build progress table as loop progresses (for inclusion in GitHub comments)

`<step name="post_loop">`: After loop completes:
- Build final results table (all issues, status, PR numbers, timing)
- If ALL issues completed (pipeline_stage == 'done'):
  - Close GitHub milestone (github.md pattern)
  - Create draft release (github.md pattern) with summary body: milestone name, issues completed count, PR links, any failures, stats
  - Advance `current_milestone` in project.json (state.md pattern)
  - Display completion banner and results table
  - Check if next milestone exists: if yes, offer auto-advance (if no failures in current)
- If some issues failed:
  - Display results table with failures highlighted
  - DO NOT close milestone or advance pointer
  - Display: "Milestone incomplete. {N} issues failed. Resolve failures and re-run /mgw:milestone."
- Post final results table as GitHub comment on a representative issue (first issue or any open one)

`<step name="report">`: Display terminal summary banner matching MGW UI patterns.

**Progress table format for GitHub comments:**
Every comment posted during milestone orchestration must include:
```markdown
**Issue #N — {Status}** {symbol}

{Status-specific detail (PR link, failure reason, etc.)}

<details>
<summary>Milestone Progress ({done}/{total} complete)</summary>

| # | Issue | Status | PR | Stage |
|---|-------|--------|----|-------|
| N | title | ✓ Done / ✗ Failed / ○ Pending / ◆ Running / ⊘ Blocked | #PR or — | pipeline_stage |

</details>
```
